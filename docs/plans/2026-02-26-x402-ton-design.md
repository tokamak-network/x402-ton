# x402-TON: HTTP 402 Payments with Native TON on Thanos Sepolia

> **SUPERSEDED**: This design was replaced by the USDC/EIP-3009 migration. See [2026-02-27-x402-usdc-migration-design.md](./2026-02-27-x402-usdc-migration-design.md) for the current design.

**Date**: 2026-02-26
**Status**: Superseded

## Overview

Full x402-compliant payment platform using **native TON** (Tokamak Network's gas token) on Thanos Sepolia. Custom `exact-ton` scheme using `msg.value` for native token transfers. Self-hosted facilitator. ERC-4337 gasless option. Reusable monorepo.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Payment token | Native TON (not WTON/ERC-20) | User preference, simpler UX, no wrapping/approvals |
| Payment mechanism | `msg.value` (not EIP-3009) | TON is native token, can't do ERC-20 approve/transferFrom |
| Facilitator | Self-hosted Express service | Coinbase facilitator doesn't support Thanos Sepolia |
| Gasless | ERC-4337 via existing Dust infra | Reuses EntryPoint + DustPaymaster already deployed |
| Architecture | Monorepo with publishable packages | Reusable by anyone on Thanos |

## Architecture

```
Client SDK                    Resource Server              Facilitator Service
(@x402-ton/client)           (@x402-ton/server)           (@x402-ton/facilitator)

GET /resource ──────────────► 402 + PAYMENT-REQUIRED
                              (scheme: "exact-ton",
                               network: "eip155:111551119090",
                               asset: "native",
                               maxAmountRequired: "1000000000000000000")

Sign EIP-712 auth ◄──────── parse requirements

GET /resource ──────────────► has PAYMENT-SIGNATURE?
+ PAYMENT-SIGNATURE header    ├─► POST /verify ──────────► check sig, nonce, deadline
                              │   ◄── { isValid: true }
                              │
                              ├─► execute handler
                              │
                              ├─► POST /settle ──────────► submit tx:
                              │   ◄── { success, txHash }   TonFacilitator.settle{value: amount}(auth, sig)
                              │                              → forwards TON to payTo
                              │                              → emits PaymentSettled
                              ▼
                              200 + PAYMENT-RESPONSE
```

## Smart Contract: TonPaymentFacilitator.sol

Deployed on Thanos Sepolia. Single contract handles settlement.

```solidity
contract TonPaymentFacilitator {
    bytes32 public immutable DOMAIN_SEPARATOR;
    mapping(bytes32 => bool) public usedNonces;

    bytes32 constant PAYMENT_AUTH_TYPEHASH = keccak256(
        "PaymentAuth(address from,address to,uint256 amount,uint256 deadline,bytes32 nonce)"
    );

    event PaymentSettled(
        address indexed from,
        address indexed to,
        uint256 amount,
        bytes32 nonce,
        bytes32 indexed paymentId
    );

    // Facilitator service calls this with native TON as msg.value
    function settle(
        address from,    // payer (signer)
        address to,      // resource server (payTo)
        uint256 amount,
        uint256 deadline,
        bytes32 nonce,
        bytes calldata signature
    ) external payable {
        require(block.timestamp <= deadline, "Expired");
        require(!usedNonces[nonce], "Nonce used");
        require(msg.value == amount, "Wrong value");

        // Verify EIP-712 signature
        bytes32 structHash = keccak256(abi.encode(
            PAYMENT_AUTH_TYPEHASH, from, to, amount, deadline, nonce
        ));
        bytes32 digest = keccak256(abi.encodePacked("\x19\x01", DOMAIN_SEPARATOR, structHash));
        address signer = ECDSA.recover(digest, signature);
        require(signer == from, "Invalid signature");

        // Mark nonce used (CEI pattern)
        usedNonces[nonce] = true;

        // Forward native TON to recipient
        (bool success, ) = to.call{value: amount}("");
        require(success, "Transfer failed");

        emit PaymentSettled(from, to, amount, nonce, digest);
    }

    // Verify without settling (view function for /verify endpoint)
    function verify(
        address from,
        address to,
        uint256 amount,
        uint256 deadline,
        bytes32 nonce,
        bytes calldata signature
    ) external view returns (bool valid, string memory reason) {
        if (block.timestamp > deadline) return (false, "Expired");
        if (usedNonces[nonce]) return (false, "Nonce used");

        bytes32 structHash = keccak256(abi.encode(
            PAYMENT_AUTH_TYPEHASH, from, to, amount, deadline, nonce
        ));
        bytes32 digest = keccak256(abi.encodePacked("\x19\x01", DOMAIN_SEPARATOR, structHash));
        address signer = ECDSA.recover(digest, signature);

        if (signer != from) return (false, "Invalid signature");
        if (from.balance < amount) return (false, "Insufficient balance");

        return (true, "");
    }

    receive() external payable {}
}
```

Key properties:
- **Replay protection**: nonce mapping, each nonce single-use
- **Deadline**: time-bound authorizations
- **CEI pattern**: state change before external call
- **On-chain verify**: facilitator can call `verify()` as view function (no gas)
- **No owner/admin**: immutable, no upgrade, no pause, no fee extraction

## Payment Flow Detail

### Standard Flow (user has TON for gas)

```
1. Client: GET /api/data
2. Server: 402 + PAYMENT-REQUIRED header (base64 JSON)
   {
     "version": 2,
     "accepts": [{
       "scheme": "exact-ton",
       "network": "eip155:111551119090",
       "maxAmountRequired": "1000000000000000000",  // 1 TON in wei
       "asset": "native",
       "payTo": "0xServerAddress",
       "maxTimeoutSeconds": 60,
       "resource": "/api/data",
       "description": "1 TON for API access"
     }]
   }
3. Client: generate random nonce, compute deadline = now + 60s
4. Client: sign EIP-712 message { from, to, amount, deadline, nonce }
5. Client: GET /api/data + PAYMENT-SIGNATURE header (base64 JSON)
   {
     "x402Version": 2,
     "scheme": "exact-ton",
     "network": "eip155:111551119090",
     "payload": {
       "signature": "0x...",
       "authorization": {
         "from": "0xClientAddress",
         "to": "0xServerAddress",
         "amount": "1000000000000000000",
         "deadline": "1740600060",
         "nonce": "0xrandom32bytes"
       }
     }
   }
6. Server → Facilitator: POST /verify { paymentPayload, paymentRequirements }
7. Facilitator: calls contract.verify() (view, no gas) → { isValid: true }
8. Server: executes handler, returns response
9. Server → Facilitator: POST /settle { paymentPayload, paymentRequirements }
10. Facilitator: submits tx: contract.settle{value: amount}(from, to, amount, deadline, nonce, sig)
    - Facilitator wallet must hold TON to fund the msg.value
    - Facilitator is reimbursed: client's signed auth proves intent to pay
    - NOTE: Facilitator fronts the TON, then gets reimbursed from... wait.
```

### Critical Design Issue: Who Holds the TON?

With EIP-3009 (standard x402), the facilitator calls `transferWithAuthorization` which pulls tokens FROM the client's wallet. The facilitator never holds user funds.

With native `msg.value`, someone must SEND the TON. Two options:

**Option A — Client pre-deposits into contract**:
- Client calls `deposit()` sending TON to the facilitator contract
- Contract tracks `balances[client]`
- On settle, contract deducts from balance and sends to `payTo`
- Pro: Facilitator never touches funds. Trustless.
- Con: Extra tx for deposit.

**Option B — Client sends TON in the payment tx**:
- Payment flow includes a tx hash where client sent TON to the contract
- Facilitator verifies the deposit happened, then calls `settle()` to release to payTo
- Pro: Single client tx. Clean.
- Con: Need to handle timing (deposit must happen before settle).

**Chosen: Option A (pre-deposit model)** — most trustless, cleanest separation.

### Revised Contract Design

```solidity
contract TonPaymentFacilitator {
    mapping(address => uint256) public balances;
    mapping(bytes32 => bool) public usedNonces;
    bytes32 public immutable DOMAIN_SEPARATOR;

    function deposit() external payable {
        balances[msg.sender] += msg.value;
        emit Deposited(msg.sender, msg.value);
    }

    function withdraw(uint256 amount) external {
        require(balances[msg.sender] >= amount, "Insufficient balance");
        balances[msg.sender] -= amount;
        (bool ok, ) = msg.sender.call{value: amount}("");
        require(ok, "Transfer failed");
        emit Withdrawn(msg.sender, amount);
    }

    function settle(
        address from, address to, uint256 amount,
        uint256 deadline, bytes32 nonce, bytes calldata signature
    ) external {
        require(block.timestamp <= deadline, "Expired");
        require(!usedNonces[nonce], "Nonce used");
        require(balances[from] >= amount, "Insufficient balance");

        // Verify EIP-712 signature
        bytes32 digest = _hashTypedData(from, to, amount, deadline, nonce);
        require(ECDSA.recover(digest, signature) == from, "Invalid signer");

        usedNonces[nonce] = true;
        balances[from] -= amount;

        (bool ok, ) = to.call{value: amount}("");
        require(ok, "Transfer failed");

        emit PaymentSettled(from, to, amount, nonce, digest);
    }

    function verify(
        address from, address to, uint256 amount,
        uint256 deadline, bytes32 nonce, bytes calldata signature
    ) external view returns (bool valid, string memory reason) {
        if (block.timestamp > deadline) return (false, "Expired");
        if (usedNonces[nonce]) return (false, "Nonce used");
        if (balances[from] < amount) return (false, "Insufficient balance");

        bytes32 digest = _hashTypedData(from, to, amount, deadline, nonce);
        if (ECDSA.recover(digest, signature) != from) return (false, "Invalid signer");

        return (true, "");
    }
}
```

### ERC-4337 Gasless Flow

For users without TON for gas:
1. Client signs EIP-712 payment authorization (off-chain, free)
2. Facilitator builds a UserOp:
   - `sender`: StealthAccount (from Dust Protocol)
   - `callData`: TonPaymentFacilitator.deposit() or direct settle
   - `paymasterAndData`: DustPaymaster signature
3. Facilitator submits via `entryPoint.handleOps()`
4. DustPaymaster covers gas

Requires: user has a StealthAccount with TON balance on Thanos Sepolia.
